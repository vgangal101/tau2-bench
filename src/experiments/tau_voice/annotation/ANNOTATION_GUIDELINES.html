1. Overview

Purpose

This document provides guidelines for annotating failed voice agent simulations to identify the first critical error that caused task failure. The goal is to characterize failure modes in voice-based customer service agents and distinguish between agent errors and user simulator artifacts.

Scope

Annotators will review failed simulations from the Tau-Voice benchmark, where a voice agent interacted with a simulated user to complete customer service tasks (e.g., returns, order modifications, account lookups).

Key Objective

For each failed simulation, identify:

  1. Error Source: Who caused the first critical error (Agent or User Simulator)?
  2. Error Type: What category of error occurred?
  3. Notes: Brief description of what went wrong.

2. Understanding the Benchmark

The Two Participants

Each simulation involves two AI participants:

The Agent (What We're Evaluating)

The Agent is a voice-based AI customer service representative from providers like OpenAI, Google, or xAI. The agent:

  • Receives audio input from the user (speech)
  • Produces audio output to the user (speech)
  • Has access to tools to look up and modify data (orders, products, user accounts)
  • Must follow a policy that defines what actions are allowed and required

The agent's job is complex: it must understand spoken language (including accents, noise, disfluencies), reason about the user's needs, and execute the correct tool calls to fulfill the request—all while following strict policy guidelines.

The User Simulator (Simulated Customer)

The User Simulator is an LLM that plays the role of a customer calling the agent. The user simulator:

  • Follows specific task instructions that define what they want to accomplish
  • Speaks naturally with disfluencies ("um", "uh"), restarts, and natural pauses
  • Only knows what's in their instructions (cannot make up information)
  • Must complete their goal and then end the conversation with ###STOP###

The user's job is intentionally simpler (by design). They just need to communicate their request clearly and follow their instructions. The burden of work is on the agent.

Why This Asymmetry Matters for Annotation

Because the agent has the complex job (policy compliance, tool usage, reasoning), agent errors are more common and expected. User simulator errors are rarer but do occur when the LLM:

  • Makes up information not in its instructions (hallucination)
  • Forgets part of its task
  • Ends the call prematurely
Key Insight: If the user clearly communicated their need and the task failed, the agent likely made the error.

3. How Tasks Are Defined

Each task has three main components that you'll see when reviewing simulations:

3.1 User Instructions (What the User Wants)

The user simulator receives structured instructions with:

FieldDescriptionExample
domainThe domain (retail, airline, telecom)"retail"
reason_for_callThe specific goal"You received order #W2378156 and wish to exchange the keyboard..."
known_infoInformation the user knows"You are Yusuf Rossi in zip code 19122"
unknown_infoInformation the user doesn't know"You do not remember your email address"
task_instructionsBehavioral guidance"You are detail-oriented and want to address everything in one go"

3.2 Agent Policy (Rules the Agent Must Follow)

The agent has a detailed policy document that defines:

Authentication Requirements:

  • Must authenticate user by email OR name + zip code
  • Must authenticate before providing any account information

Allowed Actions:

  • Cancel or modify pending orders
  • Return or exchange delivered orders
  • Modify user's default address
  • Provide information about profile, orders, products

Required Confirmations:

  • Before any action that updates the database, must list details and get explicit "yes" confirmation
"Before taking any action that updates the database, you must list the action details and obtain explicit user confirmation (yes) to proceed."

3.3 User Simulator Guidelines (How to Act Natural)

The user simulator follows voice call guidelines that instruct it to:

Speak Naturally:

  • Use disfluencies: "um", "uh", "you know"
  • Spell out special characters: "john underscore doe at gmail dot com"
  • Separate letters/numbers: "J, O, H, N" not "JOHN"

End Appropriately:

  • Generate ###STOP### when task is complete
  • Never make up information not in instructions

4. How Success Is Determined

The Key Insight: Unlike subjective evaluations, Tau-Voice success is objectively verifiable. This is what makes the benchmark reliable—and what makes your annotation job easier.

4.1 Evaluation Criteria

Each task defines exactly what must happen for success:

Expected Actions (Tool Calls): The benchmark specifies the exact tool calls the agent should make with specific arguments.

Database Checks: The benchmark verifies that the database state is correct after the conversation.

4.2 What This Means for Annotators

You don't need to understand the policy in full detail. The benchmark already determined the task failed—your job is to find why it failed.

The user's task is always simple to verify:

  • Did they communicate their known information correctly?
  • Did they follow their task instructions?
  • Did they end the call appropriately?

The agent's task is complex but you can focus on:

  • Did it understand what the user said (transcription)?
  • Did it respond at all (VAD/unresponsive)?
  • Did it make the right tool calls?
  • Did it reason correctly about the request?

5. Annotation Task

What You're Annotating

Each annotation item is a failed simulation—a conversation where the agent did not successfully complete the required task(s).

Your Goal

Identify the first critical error: the earliest point in the conversation where an error occurred that made task completion impossible or significantly more difficult.

What "Critical" Means

A critical error is one that:

  • Directly caused task failure, OR
  • Made subsequent task completion impossible/improbable, OR
  • Violated important policy/security requirements
Minor errors that don't affect the outcome should be noted but are NOT the primary focus.

6. Error Source Attribution

For each failed simulation, attribute the first critical error to one of:

Agent

The voice AI agent caused the error. Examples:

  • Agent misunderstood user speech (transcription error)
  • Agent made a reasoning/logical error
  • Agent failed to respond (VAD/unresponsive)
  • Agent called tools with wrong arguments
  • Agent violated policy (e.g., disclosed info without authentication)

User Simulator

The simulated user caused the error. Examples:

  • User hallucinated information not in their instructions
  • User didn't follow their task instructions
  • User ended the call prematurely without confirmation
  • User provided contradictory information

Mixed/Unclear

Use sparingly when:

  • Both parties contributed to the failure
  • The cause is genuinely ambiguous
Even if multiple errors occurred, focus on the first critical error that initiated the failure cascade.

8. Annotation Workflow

Follow these steps for each simulation:

  1. Review the Conversation Summary — Start with the LLM judge's summary (if available) to get an overview. Caution: The LLM judge makes mistakes—use it as a starting point only.
  2. Understand the User's Goal — Read the User Instructions to understand what they were trying to accomplish. Focus on: reason_for_call, known_info, unknown_info.
  3. Check the Evaluation Criteria — Look at the Expected Actions to see what tool calls should have happened. Check which actions succeeded vs failed.
  4. Locate the Failure Point — Review the conversation chronologically. Ask: "At what point did the task become impossible to complete?"
  5. Examine the Tick/Turn View — For voice (full-duplex) simulations, use the tick view which shows timestamps, user speech, agent speech, tool calls, and silence periods.
  6. Verify the Error — Confirm the identified error actually prevented success. If unclear from transcription, listen to the audio.
  7. Assign Attribution and Type — Determine if Agent or User caused the first critical error, select the error type, and write a brief note.

9. Edge Cases and Disambiguation

Transcription vs. Logical Error

  • Transcription: Agent's speech-to-text produced wrong text → Agent acted on wrong text
  • Logical: Agent received correct text → Agent made wrong decision

How to tell the difference: Check if the agent's transcription matches what the user actually said. If wrong, it's transcription error. If correct but agent did wrong thing, it's logical error.

VAD vs. Unresponsive vs. Input Collection

  • VAD: Agent failed to detect that user was speaking
  • Unresponsive: Agent detected speech but didn't respond
  • Input Collection: Agent heard and responded, but failed to properly gather information

User Hung Up vs. Agent Error

If the user ends the call prematurely, ask:

  1. Did the agent give any indication the task was complete?
  2. Was the agent unresponsive, forcing the user to give up?
  3. Did the user reasonably believe the task was done?

If the agent's behavior caused the user to reasonably conclude the task was complete → Agent Error

Multiple Errors

If multiple errors occurred:

  • Focus on the first critical error
  • Note subsequent errors in the Notes field if relevant
  • Ask: "If the first error hadn't happened, would the task have succeeded?"

10. Output Format

For each simulation, record:

ReviewerYour name/identifierString

Error TypeCategory of the errorSee taxonomy above
NotesBrief explanation of what happenedFree text

Notes Field Guidelines

  • Keep notes concise (1-3 sentences)
  • Include specific details: tick ranges, tool names, exact phrases when relevant
  • Mention if the LLM judge's assessment was correct/incorrect
  • Flag any uncertainty or ambiguity

11. Examples

Example 1: Agent Transcription Error

User Instructions: Exchange keyboard for one with clicky switches. Known: "Yusuf Rossi, zip 19122"

What happened: User spelled out their zip code as "one, nine, one, two, two". Agent transcribed as "19112" instead of "19122". Authentication failed repeatedly.

Error Source: Agent
Error Type: transcription
Notes: Agent mistranscribed zip code despite user spelling it out clearly.

Example 2: Agent Logical Error

User Instructions: Cancel all pending orders.

What happened: Agent correctly identified 3 pending orders via tool call, then said "I see you have 2 pending orders" and only cancelled 2.

Error Source: Agent
Error Type: logical
Notes: Agent tool result showed 3 pending orders but agent claimed only 2 exist.

Example 3: Agent VAD/Unresponsive

User Instructions: Exchange an item in a delivered order.

What happened: User provided all exchange details and confirmed "yes". Agent went silent for 25 seconds. User checked in twice, no response. User hung up frustrated.

Error Source: Agent
Error Type: VAD/unresponsive
Notes: Ticks 1:45-2:10. Agent disappeared after user spelled info.

Example 4: User Hallucination

User Instructions: "Return the jacket." (no size specified in instructions)

What happened: User told agent "I want to return my size L jacket" when the actual order was size M.

Error Source: User
Error Type: hallucination
Notes: User hallucinated jacket size (L) not in instructions.

Example 5: User Instruction Following Error

User Instructions: "Cancel pending orders AND ask about return policy."

What happened: User successfully cancelled orders, then said goodbye without asking about return policy.

Error Source: User
Error Type: instruction_following
Notes: User completed first task but forgot second part of instructions.

Example 6: Agent Tool Call Error

User Instructions: Look up order status. Known: "Name: Sofia Chen"

What happened: Agent collected name correctly but called find_user_by_name with argument "first_name": "S-O-F-I-A" including the hyphens.

Error Source: Agent
Error Type: tool_call
Notes: Agent included literal hyphens in tool call argument. Should have parsed spelled-out letters into "Sofia".

Quick Reference Card

When Attributing to Agent:

  • Transcription was wrong (check logs)
  • Agent went silent / stopped responding
  • Tool call had wrong arguments
  • Agent made incorrect logical inference
  • Agent violated policy

When Attributing to User:

  • User said information NOT in their instructions
  • User forgot part of their task
  • User ended call without confirmation when agent was still working
Default Assumption: If user clearly communicated their goal and the task failed → likely agent error

Questions?

If you encounter cases not covered by these guidelines, document them and flag for discussion. Edge cases will be addressed through team calibration sessions.
